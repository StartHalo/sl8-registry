# Frame-driven 3D in Remotion (`@remotion/three`) — DEFERRED contract

> **Status: DEFERRED (REQ-005 RAM tier).** This document is the **activation contract**, not a v1
> authoring guide. In v1, `rm-build`'s composition-contract rule **C9** forbids `@remotion/three` and
> `rm-validate` rejects it (H-RAM → Exit-137 on the ~1.9 GB sandbox). Do not author 3D until REQ-005
> raises the runtime RAM tier. Ported and re-targeted from the bundled official rule
> `remotion-best-practices/rules/3d.md`.

Follow React Three Fiber and Three.js best practices. Only the **Remotion-specific** rules below are
load-bearing — they are what makes 3D render deterministically under the headless frame-stepping renderer
instead of flickering or freezing.

## 0. RAM reality (why this is gated)
Three.js renders WebGL **every frame**; it is the heaviest surface in `research/capabilities-catalog.md`
(C7/C8/C9, all **H-RAM**). The sandbox renders at `--concurrency=1` against a ~1.9 GB ceiling; a 3D scene
pushes past it → **Exit-137** (OOM). REQ-005 (the RAM tier) is the gate. Even after it lands: ONE canvas,
low-poly geometry, short duration, ≤1080p.

## 1. Prerequisites
Install `@remotion/three` **per-project**, pinned to the one Remotion version the runtime ships (every
`@remotion/*` resolves to the same version — `init.sh`/`render.sh` enforce the pin):

```bash
npx remotion add @remotion/three   # pulls peer deps three + @react-three/fiber
```

## 2. Wrap 3D in `<ThreeCanvas>` with real lighting
You MUST wrap all 3D content in `<ThreeCanvas>`, and it MUST receive `width` and `height` (from
`useVideoConfig()`). Without lighting, `meshStandardMaterial` renders black.

```tsx
import { ThreeCanvas } from "@remotion/three";
import { useVideoConfig, useCurrentFrame } from "remotion";

export const Scene3D: React.FC = () => {
  const { width, height } = useVideoConfig();
  const frame = useCurrentFrame();
  const rotationY = frame * 0.02; // frame-driven, NOT wall-clock

  return (
    <ThreeCanvas width={width} height={height}>
      <ambientLight intensity={0.4} />
      <directionalLight position={[5, 5, 5]} intensity={0.8} />
      <mesh rotation={[0, rotationY, 0]}>
        <boxGeometry args={[2, 2, 2]} />
        <meshStandardMaterial color="#4a9eff" />
      </mesh>
    </ThreeCanvas>
  );
};
```

## 3. Animate with `useCurrentFrame()` only — `useFrame()` is FORBIDDEN
Shaders, models, cameras, and materials **MUST NOT** animate by themselves. **No** animation is allowed
unless it is driven by `useCurrentFrame()`. Self-animating 3D (or `useFrame()` from `@react-three/fiber`)
flickers during render because the renderer steps discrete frames headless — wall-clock motion renders as
a frozen or torn frame.

```tsx
const frame = useCurrentFrame();
const rotationY = frame * 0.02;
<mesh rotation={[0, rotationY, 0]}>…</mesh>;
```

This is the 3D form of `rm-build`'s contract rule **C1** (frame-driven only): everything reads
`useCurrentFrame()`/`interpolate`/`spring`, nothing reads the clock.

## 4. `<Sequence>` inside `<ThreeCanvas>` MUST be `layout="none"`
A `<Sequence>` rendered inside the canvas has no DOM layout box, so set `layout="none"` or it errors.

```tsx
import { Sequence } from "remotion";
import { ThreeCanvas } from "@remotion/three";

<ThreeCanvas width={width} height={height}>
  <Sequence layout="none">
    <mesh>
      <boxGeometry args={[2, 2, 2]} />
      <meshStandardMaterial color="#4a9eff" />
    </mesh>
  </Sequence>
</ThreeCanvas>;
```

## 5. Determinism (rm-build contract C3)
Any randomness (particle fields, jitter) uses the engine's seeded helpers (`noise(seed, frame, salt)` /
`random("<seed>")`) with a fixed `seed` — **never** `Math.random`. The vision grade is meaningless if
frames vary run-to-run.
