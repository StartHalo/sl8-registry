# rm-voiceover — narration audio + word-level timing (Remotion)

## Purpose
Turn the beat-structured `02-script.md` into the project's **spoken audio** (`assets/vo/narration.wav`, one
continuous track) and the **word-level timing** (`04-timing.json`) that `rm-build` (caption blocks, beat clip
timing, `durationInFrames`) and `rm-render` (audio mux) consume. Both come through the keyless **`ai-gen`**
SL8 proxy — the full narration is synthesized in **one** TTS call (Kokoro), then transcribed in **one** ASR
pass (Wizper) and the recovered per-word start/end times are split back into beats by order. (Batching
collapses what used to be one TTS + one ASR call *per beat* — 2×N serial round-trips — down to 2.) This skill
never authors React/HTML and never renders video; it only produces the audio + timing assets. For a
**clip-in** job (JTBD-3) it runs **ASR-only** on the input clip — no TTS, no `narration.wav`.

`04-timing.json` is **`@remotion/captions`-shaped already**, in **seconds**; `rm-captions`/`rm-build` do the
×1000 → `startMs`/`endMs` conversion (see `references/ai-gen-audio.md` §5). Keep times in seconds here.

`$SKILL` = this skill's directory. Confirmed commands, voices, the `-o-is-a-directory` gotcha, the
`@remotion/captions` mapping, and the JTBD-3 ASR-entry recipe live in `references/ai-gen-audio.md`.

## When to use
- **Voiceover** (phase 4, JTBD-1): after `rm-script` wrote `02-script.md`, before `rm-build`.
- **Re-voice** (JTBD-4): a follow-up like "use a different voice" / "re-voice with af_nova" — re-run from
  the SAME `02-script.md` (the narration text must NOT change; only the voice changes). Facts only change by
  re-running `rm-script` (phase 2).
- **Clip → captions, ASR entry** (JTBD-3): no script. Transcribe the input clip's audio to recover word
  timings for `rm-captions` (no TTS, no `narration.wav`). See step 3b and `references/ai-gen-audio.md` §6.
- Do NOT use to write the script (that is `rm-script`), to build the composition (`rm-build` / `rm-captions`),
  or to produce cutouts/captures/mattes (`rm-assets`).

## Inputs
- `artifacts/<project>/02-script.md` (required for JTBD-1/4) — the per-beat script. Each beat has a VO line;
  those lines, in order, are the narration. If it is missing/empty *and this is not a clip-in job*, record the
  failure in `state.md` and stop (run `rm-script` first) — do NOT invent narration.
- `artifacts/<project>/assets/captures/<clip>` (JTBD-3) — the input video/audio clip to transcribe.
- `voice` (optional) — resolve from the user request → else `context.md` (the onboarding voice default) →
  else `am_michael`. State which you used.
- `model` (optional) — default `fal-ai/kokoro/american-english`.

Read these before writing anything (READ-BEFORE-WRITE).

## Instructions

### 1. Extract the beat narration from the script (JTBD-1/4)
Read `02-script.md` and pull the **VO / narration line for each beat in order** (ignore the on-screen-text
and direction lines). Build a small list of `(beat_id, narration_text)` — `beat_id` is a stable kebab slug
(`beat-01`, `beat-02`, … or the script's own beat labels). This is the only place narration comes from;
never add or paraphrase sentences the script doesn't contain.

If the script has a single continuous narration with no explicit beats, treat each sentence (or each
storyboard scene, if `03-storyboard.md` exists) as a beat.

Then write that list to a **beats-tsv** — one line per beat, `beat_id<TAB>narration_text`, in order — and
**join** the narration lines in order into one continuous narration string (a single space between lines).
The beats-tsv records the beat boundaries; the joined string is what gets synthesized.
```bash
mkdir -p artifacts/<project>/work artifacts/<project>/assets/vo
# beats.tsv: one line per beat — beat_id<TAB>narration_text (in order)
printf 'beat-01\t%s\n' "Ship faster."     >  artifacts/<project>/work/beats.tsv
printf 'beat-02\t%s\n' "Build it today."  >> artifacts/<project>/work/beats.tsv
# … one printf per beat. The joined narration = those texts in order, space-separated.
```
(For a **clip-in** job, skip steps 1–2 and go to step 3b.)

### 2. Synthesize the voiceover (ONE TTS call → narration.wav)
Synthesize the **whole narration in a single TTS call** with the `tts.sh full` mode. **`-o` is a
DIRECTORY**, so the wrapper targets a temp dir and captures the produced wav via `ls`, then writes it to
`assets/vo/narration.wav`:
```bash
bash "$SKILL/scripts/tts.sh" full "<joined-narration>" artifacts/<project>/assets/vo "<voice>" "<model>"
#                            ^mode  ^all VO lines joined  ^out-dir                       ^voice  ^model(optional)
```
`tts.sh full` runs `ai-gen audio tts "<text>" -m <model> -o <tmpdir>` ONCE into a fresh temp directory, finds
the single produced wav there (`ls <tmpdir>/*.wav`), and moves it to
`artifacts/<project>/assets/vo/narration.wav`. It prints the final path and exits non-zero only if no wav was
produced (→ take the silent fallback, step 4). The Kokoro voice is selected by passing the voice id (see
`references/ai-gen-audio.md` for how the voice maps onto the `--voice` argument). **Do not** call `ai-gen`
directly; always go through `tts.sh` so the directory-capture stays correct. (The legacy per-beat synth mode
and `tts.sh ... concat` remain available as a fallback, but the batch `full` mode is the default.)

### 3. Recover word-level timing (ONE ASR pass → split into beats)
Transcribe `narration.wav` **once** and split the word-timings back into beats with `words.sh full`, which
calls `ai-gen audio stt "<narration.wav>" -m fal-ai/wizper`, parses the JSON for per-word `start`/`end` (when
the model returns them), splits the flat word stream into beats **by order** using the beats-tsv, and writes
`04-timing.json`:
```bash
bash "$SKILL/scripts/words.sh" full artifacts/<project>/assets/vo/narration.wav \
                                    artifacts/<project>/04-timing.json \
                                    artifacts/<project>/work/beats.tsv
#                              ^mode ^single narration wav      ^timing output         ^beats-tsv (beat boundaries)
```
`words.sh full` transcribes once, then **walks the beats in order**, giving each beat the next slice of the
flat word stream sized to that beat's token count. If the Wizper word count differs from the script token
count, it splits **proportionally** by each beat's fraction of the total tokens. Each beat's `start` = its
first word's start, `end` = its last word's end; the flat `words[]` track keeps **absolute** times tagged
with `beat`. It emits (times in **seconds**):
```json
{
  "beats": [
    { "beat": "beat-01", "wav": "assets/vo/narration.wav", "duration": 0.84, "timing_method": "wizper",
      "words": [ { "text": "Ship", "start": 0.00, "end": 0.31 }, { "text": "faster", "start": 0.31, "end": 0.84 } ] }
  ],
  "words": [ { "text": "Ship", "start": 0.00, "end": 0.31, "beat": "beat-01" } ],
  "total_duration": 12.40,
  "source": "ai-gen audio stt fal-ai/wizper"
}
```
Every beat's `wav` is `assets/vo/narration.wav` (the single track). The flat `words[]` track is **absolute
timeline time** (what `rm-build`'s caption blocks read, and what `rm-captions` maps ×1000 → `@remotion/captions`
`Caption[]`). If Wizper returns only the transcript text with no per-word timestamps, `words.sh full` **falls
back to even distribution** (`"timing_method": "even"` — approximate but never fabricated). If STT fails
entirely, it emits beats from the beats-tsv with **estimated** pacing (`max(0.8, tokens/2.5)` s per beat,
`"timing_method": "estimated"`, `words: []`).

### 3b. Clip-in: ASR-only entry (JTBD-3, no TTS)
There is no `02-script.md`. Transcribe the **input clip's own audio** to recover word timings for
`rm-captions`; do not synthesize anything and do not write `narration.wav`. Extract a mono 16 kHz wav from a
video first (Wizper wants audio; bash 3.2, no `timeout`), use a **single-beat** beats-tsv so the whole
transcript lands in one beat, then run the same `words.sh full`:
```bash
mkdir -p artifacts/<project>/work
ffmpeg -v error -y -i artifacts/<project>/assets/captures/<clip>.mp4 \
       -vn -ac 1 -ar 16000 artifacts/<project>/work/clip-audio.wav      # skip if the input is already audio
printf 'clip\t.\n' > artifacts/<project>/work/beats.tsv                  # one beat absorbs the whole transcript
bash "$SKILL/scripts/words.sh" full artifacts/<project>/work/clip-audio.wav \
                                    artifacts/<project>/04-timing.json \
                                    artifacts/<project>/work/beats.tsv
```
The flat `words[]` is absolute clip-timeline time — exactly what `rm-captions` overlays on the
`<OffthreadVideo>`. The clip itself is the audio at render time; there is no `narration.wav` to mux. Record
"JTBD-3 ASR-only (no TTS)" in `state.md`. If ASR is unreachable, `words.sh full` writes estimated pacing
(`words:[]`) — note that captions can't word-sync and let `rm-captions` fall back to line/chunk captions.

### 4. Headless fallback (TTS unreachable → render silent)
If `tts.sh full` cannot produce `narration.wav` (ai-gen TTS unreachable / `success:false`, non-zero exit),
do NOT fail the project and do NOT prompt the user:
- Skip the audio, and still write `04-timing.json` by running `words.sh full` anyway — point it at the
  (missing) `narration.wav`; with no wav it falls back to **script-estimated** pacing
  (≈ `max(0.8, tokens/2.5)` seconds per beat, the legibility pacing rule) with `"timing_method": "estimated"`
  and `"words": []` for every beat. (The timing track still paces the beats and gives `rm-build` a duration.)
- Record in `state.md` (phase 4) that the voiceover fell back to **silent** (why), so `rm-build` derives
  `durationInFrames` from the estimated total and `rm-render` knows there is no audio stream to mux and the
  final summary reports it.
- This is the JTBD-1 documented fallback: "Kokoro/voice unavailable → render silent + note it" — the video
  still ships; the timing track still drives beat pacing.

### 5. Report
State the resolved **voice** and **model** used, how many beats got real TTS audio vs fell back to silent (or,
for JTBD-3, that this was ASR-only on the input clip), the total narration duration, and whether word timings
are real (Wizper word-level) or estimated/even. Mark `state.md` phase 4 done (or `blocked`/fallback-noted),
and remember.

## Outputs
- `artifacts/<project>/assets/vo/narration.wav` — one continuous narration track for the whole project,
  generated by a single `ai-gen` TTS call. (Absent when the voiceover fell back to silent, or for a JTBD-3
  clip-in job.) Per-beat wav slices are **not** produced by default — one VO track is the model. If a
  downstream skill needs per-beat slices, cut them locally from `narration.wav` using the beat boundaries
  (`start`/`duration`) in `04-timing.json` via `ffmpeg` (`-ss <start> -t <duration>`), no extra network calls.
- `artifacts/<project>/04-timing.json` — word-level timings (in **seconds**): a per-beat `beats[]` with
  `words[]` and a flat absolute-time `words[]` track for caption sync, plus `total_duration` and the timing
  method. This is the `@remotion/captions`-shaped timing contract `rm-build` (`durationInFrames` + caption
  blocks) and `rm-captions` (×1000 → `Caption[]`) read.

## Examples

### Example 1: narrate a 12s teaser (JTBD-1)
`02-script.md` has 5 beats. Write `work/beats.tsv` (5 lines), join the 5 VO lines, run `tts.sh full` ONCE →
`assets/vo/narration.wav`, then `words.sh full narration.wav 04-timing.json beats.tsv` ONCE → `04-timing.json`
with the 5 `beats[]` split by order and a flat `words[]`. Report: "voice `am_michael` (default), Kokoro;
narration voiced (1 TTS + 1 ASR call); 12.4s total; word-level timings from Wizper." → hand off to `rm-build`.

### Example 2: re-voice an existing project (JTBD-4)
User: "re-voice it with af_nova." Re-read the UNCHANGED `02-script.md`, set `voice=af_nova`, re-run steps 1–3
(overwriting `assets/vo/narration.wav` + `04-timing.json`). Append a dated note to `state.md` ("re-voiced
am_michael → af_nova; narration text unchanged"). Do NOT touch the script facts; `rm-render` re-muxes the new
wav (no rebuild needed unless the duration changed materially).

### Example 3: clip → captions, ASR entry (JTBD-3)
User: "add TikTok captions to this clip." No script. Extract `work/clip-audio.wav` from the staged clip, write
a single-beat `work/beats.tsv`, run `words.sh full clip-audio.wav 04-timing.json beats.tsv` → flat absolute
`words[]`. No `narration.wav`. Report "ASR-only on the input clip (no TTS); Wizper word timings recovered" →
hand the timing to `rm-captions`.

### Example 4: TTS unreachable (headless fallback)
`tts.sh full` returns `success:false` (non-zero exit). Run `words.sh full` against the missing wav → it writes
`04-timing.json` from script-estimated pacing (`timing_method:"estimated"`, `words:[]`). Record "voiceover →
silent fallback (ai-gen TTS unreachable)" in `state.md`, and report it. `rm-build`/`rm-render` proceed without
an audio stream.

## Troubleshooting
- **No wav after TTS / "where did the file go?"** — `-o` is a **directory**, not a filename. The wav lands at
  `<dir>/american-english-<ts>.wav`. `tts.sh full` already captures it via `ls <dir>/*.wav` and writes it to
  `narration.wav`; if you called `ai-gen` directly you missed this — use `tts.sh`.
- **`success:false` in the ai-gen JSON** — the model is unreachable or the call was malformed. `tts.sh full`
  exits non-zero and prints the JSON; take the silent fallback (step 4). Do not retry-loop forever.
- **`04-timing.json` has words but no timestamps** — Wizper returned text only; `words.sh full` then used even
  distribution over the wav's `ffprobe` duration, partitioned into beats by token fraction
  (`timing_method:"even"`). Captions will be approximately synced — acceptable; note it.
- **Captions land at the wrong time in the render** — `04-timing.json` is in **seconds**; `rm-captions` must
  multiply by 1000 for `@remotion/captions` (`startMs`/`endMs`). Do not pre-multiply here. See
  `references/ai-gen-audio.md` §5.
- **Beat word counts look off** — `words.sh full` splits the single transcript into beats by order: it gives
  each beat its token count of words, or, if Wizper's word count differs from the script's, a proportional
  share. Boundaries stay monotonic and every word is placed; minor drift at beat edges is expected and fine.
- **Wrong voice / accent** — pass the right Kokoro voice id (see `references/ai-gen-audio.md`); the default
  model is American-English Kokoro. A different language needs a different Kokoro model — say so, don't
  silently mistranslate.
- **`ffprobe`/`jq` absent** — `words.sh` falls back to `node` for JSON and a script-estimated duration; timings
  become estimated. Prefer running in-sandbox (`sl8-animation`) where ffmpeg/ffprobe/node are all present.

## Quality Criteria
- [ ] One non-empty `assets/vo/narration.wav` (the single continuous track) when TTS reached, or a recorded
  silent fallback — or, for JTBD-3, no `narration.wav` (ASR-only on the input clip).
- [ ] `04-timing.json` exists with a non-empty flat `words[]` track (real Wizper timings, or even/estimated
  with the method recorded), in **seconds** and monotonic.
- [ ] Narration text is exactly the script's VO lines, joined in order — nothing invented, nothing dropped;
  beats split by order. (JTBD-3: the words are exactly Wizper's transcript of the clip.)
- [ ] On a re-voice, the script facts are unchanged; only the voice differs.
- [ ] The resolved voice/model and any silent/even/estimated fallback are reported and written to `state.md`.
- [ ] Keyless via `ai-gen`; no ElevenLabs/HeyGen auth/TTS/ASR, no cloud — never prompts the user when headless.
